# GitHub Actions Release Workflowのレビュー

## 概要

GitHub Actionsのリリースワークフローファイル（release.yml）は、プロジェクトのバージョン管理とリリース自動化を行います。設定は機能的ですが、いくつかの重要な改善点が見られます。

## 現在の設定の分析

### トリガー設定

```yaml
name: Release

on:
  push:
    branches:
      - main
      - master
  workflow_dispatch:

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
```

- **✅ 良い点**:
  - mainとmasterブランチへのプッシュで自動実行
  - 手動トリガー（workflow_dispatch）のサポート
  - 同時実行の制御

- **⚠️ 改善点**:
  - タグプッシュでのトリガーがない
  - リリースブランチの考慮がない

### バージョンバンプジョブ

```yaml
jobs:
  bump-version:
    name: Bump version
    runs-on: ubuntu-latest

    outputs:
      version: ${{ steps.tag-version.outputs.new_version }}
      tag: ${{ steps.tag-version.outputs.new_tag }}
      changelog: ${{ steps.tag-version.outputs.changelog }}

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: 🏷️ Bump version and push tag
        id: tag-version
        uses: mathieudutour/github-tag-action@v6.2
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          default_bump: "minor"
          custom_release_rules: "feat:minor:✨ Features,fix:patch:🐛 Fixes,..."
```

- **✅ 良い点**:
  - セマンティックバージョニングの自動化
  - カスタムリリースルールの定義
  - 適切な出力変数の定義

- **⚠️ 改善点**:
  - バージョン履歴の保存がない
  - チェンジログの検証がない

## 改善提案

### 1. トリガー設定の拡張

```yaml
name: Release

on:
  push:
    branches:
      - main
      - master
      - 'release/**'
    tags:
      - 'v*'
  pull_request:
    types: [closed]
    branches:
      - main
      - master
  workflow_dispatch:
    inputs:
      version_type:
        description: 'Version bump type'
        required: true
        default: 'minor'
        type: choice
        options:
          - patch
          - minor
          - major

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

### 2. バージョン管理の強化

```yaml
jobs:
  version-check:
    runs-on: ubuntu-latest
    outputs:
      should_release: ${{ steps.check.outputs.should_release }}
    steps:
      - name: Check version changes
        id: check
        run: |
          if [[ "${{ github.event_name }}" == "push" && "${{ github.ref }}" =~ ^refs/tags/v ]]; then
            echo "should_release=true" >> $GITHUB_OUTPUT
          elif [[ "${{ github.event.pull_request.merged }}" == "true" ]]; then
            echo "should_release=true" >> $GITHUB_OUTPUT
          else
            echo "should_release=false" >> $GITHUB_OUTPUT
          fi

  bump-version:
    needs: version-check
    if: needs.version-check.outputs.should_release == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Get version from input
        if: github.event_name == 'workflow_dispatch'
        run: echo "VERSION_TYPE=${{ github.event.inputs.version_type }}" >> $GITHUB_ENV

      - name: Determine version bump
        if: github.event_name != 'workflow_dispatch'
        run: |
          if grep -q "BREAKING CHANGE:" .git/COMMIT_EDITMSG; then
            echo "VERSION_TYPE=major" >> $GITHUB_ENV
          elif grep -q "feat:" .git/COMMIT_EDITMSG; then
            echo "VERSION_TYPE=minor" >> $GITHUB_ENV
          else
            echo "VERSION_TYPE=patch" >> $GITHUB_ENV
          fi
```

### 3. ビルドとテストの強化

```yaml
  build-and-test:
    needs: bump-version
    runs-on: windows-latest
    strategy:
      matrix:
        configuration: [Release]
    steps:
      - uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '9.0.x'

      - name: Cache NuGet packages
        uses: actions/cache@v4
        with:
          path: ~/.nuget/packages
          key: ${{ runner.os }}-nuget-${{ hashFiles('**/*.csproj') }}

      - name: Update version
        run: |
          $version = "${{ needs.bump-version.outputs.version }}"
          (Get-Content "Directory.Build.props") -replace '<Version>.*</Version>', "<Version>$version</Version>" | Set-Content "Directory.Build.props"

      - name: Build and test
        run: |
          dotnet restore
          dotnet build --configuration ${{ matrix.configuration }} --no-restore
          dotnet test --configuration ${{ matrix.configuration }} --no-build
```

### 4. リリースの作成と成果物の公開

```yaml
  create-release:
    needs: [bump-version, build-and-test]
    runs-on: ubuntu-latest
    steps:
      - name: Create Release
        uses: softprops/action-gh-release@v1
        with:
          tag_name: ${{ needs.bump-version.outputs.tag }}
          name: Release ${{ needs.bump-version.outputs.version }}
          body: ${{ needs.bump-version.outputs.changelog }}
          draft: false
          prerelease: false
          files: |
            ./artifacts/*.exe
            ./artifacts/*.zip
          token: ${{ secrets.GITHUB_TOKEN }}
```

## セキュリティ上の考慮事項

1. **コード署名**:

```yaml
      - name: Sign executable
        uses: GIGWorkspace/code-sign-action@v1
        with:
          certificate: ${{ secrets.CODE_SIGNING_CERT }}
          password: ${{ secrets.CODE_SIGNING_PASSWORD }}
          folder: artifacts
```

2. **成果物の検証**:

```yaml
      - name: Verify artifacts
        run: |
          Get-ChildItem artifacts/*.exe | ForEach-Object {
            $hash = Get-FileHash $_.FullName -Algorithm SHA256
            $hash.Hash | Out-File -Append artifacts/checksums.txt
          }
```

## パフォーマンスの最適化

1. **ビルドキャッシュ**:

```yaml
      - name: Cache build results
        uses: actions/cache@v4
        with:
          path: |
            ~/.nuget/packages
            **/obj/project.assets.json
          key: ${{ runner.os }}-build-${{ hashFiles('**/*.csproj') }}
```

2. **並列ビルド**:

```yaml
    strategy:
      matrix:
        configuration: [Release]
        platform: [x64, x86]
      fail-fast: false
```

## 推奨事項

1. **バージョン管理の改善**:
   - セマンティックバージョニングの厳密な適用
   - バージョン履歴の保存
   - 自動チェンジログ生成の検証

2. **ビルドプロセスの強化**:
   - マルチプラットフォームビルド
   - コード署名の導入
   - 成果物の自動検証

3. **リリース管理**:
   - プレリリース/ドラフトリリースのサポート
   - リリースノートのテンプレート化
   - 成果物のチェックサム生成

4. **ドキュメント**:
   - 自動API文書生成
   - リリースノートの自動更新
   - Wikiの更新

## まとめ

現在のリリースワークフローは基本的な機能を提供していますが、以下の点での改善が推奨されます：

1. より柔軟なバージョン管理
2. 包括的なビルドとテスト
3. セキュアなリリースプロセス
4. 自動化されたドキュメント生成
5. 効率的な成果物管理
